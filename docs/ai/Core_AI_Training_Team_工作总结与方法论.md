## 1. 团队定位与核心使命

### 1.1 团队性质

Core AI Training Team 处于 AI Agent 产品链路的**后方质量保障与能力优化层**。团队不直接开发模型，而是作为模型能力的"质检官"与"优化引擎"，通过系统化的评估、基准测试与案例扩展，持续为模型迭代提供数据驱动的反馈。

### 1.2 核心职责

| 职责维度     | 具体内容                                       |
| ------------ | ---------------------------------------------- |
| **能力评估** | 多模型在真实场景下的横向对比评测               |
| **基准工程** | Benchmark 研究、Pipeline 构建、案例扩展        |
| **质量守护** | 通过 Build + Sanity Check 闭环保障交付物有效性 |
| **洞察提炼** | 从测试中发现模型短板，提炼优化方向             |

### 1.3 在产品链路中的位置

```
Model Training → Model Release → [Core AI Training Team] → Feedback Loop → Next Iteration
                                  ├── 能力评估 & 打分排名
                                  ├── Benchmark 案例扩展
                                  ├── Pipeline 自动化验证
                                  └── 优化方向洞察报告
```

## 2. Phase 1 — 多模型能力评估与排名体系

### 2.1 评估背景

在 VS Code 集成的多模型生态中，团队需要建立统一的评估框架，对主流模型（GPT、Claude Opus、Gemini 等）进行能力摸底，形成可量化的对比基线。

### 2.2 评估维度矩阵

| 维度                          | 评估焦点                         | 典型场景                              |
| ----------------------------- | -------------------------------- | ------------------------------------- |
| **UI Case**                   | 界面交互相关代码的理解与生成能力 | 组件渲染、样式修复、响应式布局        |
| **Bug-related**               | 缺陷定位、根因分析与修复能力     | 运行时错误、逻辑漏洞、边界条件        |
| **Feature & Improvement**     | 功能实现与代码优化建议能力       | 新功能开发、性能优化、重构建议        |
| **Documentation & Knowledge** | 技术文档生成与知识问答准确度     | API 文档、注释生成、技术解释          |
| **Security**                  | 安全漏洞识别与安全编码能力       | XSS/CSRF 检测、依赖漏洞、安全最佳实践 |

### 2.3 评估方法论

```
1. 设计标准化 Case Set（每个维度 N 个 Case）
2. 各模型在 VS Code 环境中独立执行
3. 按预定义 Rubric 打分（准确性 / 完整性 / 代码质量 / 响应速度）
4. 多轮验证取均值，消除随机性
5. 汇总为雷达图 + 综合排名表
```

### 2.4 输出交付物

- **模型能力雷达图**：各维度得分可视化
- **优缺点对照表**：各模型的强项与弱项详列
- **综合排名报告**：加权总分排名 + 各维度细分排名
- **优化建议清单**：基于弱项的改进方向建议

### 2.5 关键发现（框架示例）

| 模型        | 强项                   | 弱项                     | 综合排名 |
| ----------- | ---------------------- | ------------------------ | -------- |
| GPT         | Feature 理解、文档生成 | 复杂 UI 场景的上下文丢失 | —        |
| Claude Opus | Bug 根因分析、安全维度 | 长上下文下的响应一致性   | —        |
| Gemini      | 多语言支持、知识广度   | 代码精确度待提升         | —        |

> _具体分数为保密数据，此处仅展示分析框架。_

## 3. Phase 2 — Terminal-Bench2 深入研究与 Pipeline 构建

### 3.1 Terminal-Bench2 (TB2) 概述

Terminal-Bench2 是一套面向 **AI Agent 终端操作能力**的基准测试框架。它通过预定义的 seed cases，评估模型在真实终端环境中的任务执行能力，涵盖系统管理、文件操作、环境配置、网络调试等多个维度。

### 3.2 研究与学习路径

```
Step 1: 理解 TB2 架构
  ├── Seed Cases 的结构与定义
  ├── 评估指标与判定逻辑
  └── 执行环境的容器化方案

Step 2: Pipeline 构建
  ├── Build Pipeline — 构建测试镜像
  └── Sanity Check Pipeline — 验证案例有效性

Step 3: 脚本完善
  ├── 定义输出内容的语义
  ├── 规范化日志格式
  └── 错误处理与重试机制
```

### 3.3 Build Pipeline

**目标**：将 test case 及其依赖打包为可执行的 Docker 镜像。

```
[Case Definition] → [Dependency Resolution] → [Dockerfile Generation] → [Image Build] → [Registry Push]
```

**关键要素**：

- 每个 case 对应独立的运行环境，确保隔离性
- 基础镜像标准化，减少构建时间
- 依赖声明清晰，避免运行时缺失
- 构建产物可追溯（tag 策略 + metadata）

### 3.4 Sanity Check Pipeline

**目标**：验证构建出的镜像中，case 能够正常执行并产出预期结果。

```
[Pull Image] → [Launch Container] → [Execute Case] → [Validate Output] → [Pass/Fail Report]
```

**验证维度**：

- 环境完整性：所有依赖是否就绪
- 执行可达性：case 能否正常启动并运行到完成
- 输出一致性：结果是否符合预定义的 expected output
- 超时保护：设置合理 timeout 防止挂起

### 3.5 输出内容语义定义

| 输出字段            | 含义                                      |
| ------------------- | ----------------------------------------- |
| `status`            | 执行状态（pass / fail / timeout / error） |
| `exit_code`         | 进程退出码                                |
| `stdout`            | 标准输出内容                              |
| `stderr`            | 标准错误输出                              |
| `duration`          | 执行耗时（ms）                            |
| `validation_result` | 结果校验是否通过                          |
| `metadata`          | 环境信息、版本号等附加数据                |

## 4. Phase 3 — Sub Cases 扩展与交付验证闭环

### 4.1 Seed Case 深度分析方法论

> **核心原则**：扩展之前，必须先彻底理解。

对每一个 TB2 Seed Case 执行以下分析：

| 分析维度       | 具体内容                   | 示例                                |
| -------------- | -------------------------- | ----------------------------------- |
| **考察方向**   | 该 case 测试的是哪个能力域 | 文件系统操作 / 网络配置 / 进程管理  |
| **考察知识点** | 涉及的具体技术知识         | `chmod` 权限模型、`iptables` 规则链 |
| **考察难点**   | 哪些环节容易使模型出错     | 多步骤依赖、隐式前置条件、边界情况  |
| **前置条件**   | case 执行所需的环境假设    | 特定 OS 版本、已安装的工具链        |
| **预期行为**   | 正确完成的判定标准         | 输出匹配、文件状态变更、进程状态    |

### 4.2 Sub Case 扩展策略（Extend-K）

基于 seed case 分析结果，在**同等难度层级**上做横向扩展：

```
Seed Case (A)
  ├── 分析: 考察 "文件权限管理" → 知识点 "chmod 数字模式"
  │
  ├── Sub Case A-1: chmod 符号模式 (同层级，不同表达方式)
  ├── Sub Case A-2: chown + chmod 组合操作 (同层级，关联知识点)
  ├── Sub Case A-3: ACL 权限设置 (同层级，同域扩展)
  └── Sub Case A-K: setfacl 默认权限继承 (同层级，进阶变体)
```

**扩展原则**：

1. **同层同维**：sub case 的难度和考察深度应与 seed case 保持一致
2. **知识关联**：扩展方向应覆盖 seed case 考察知识点的相邻知识域
3. **独立可测**：每个 sub case 必须可独立执行和验证
4. **无冗余**：避免与已有 seed case 或其他 sub case 高度重复

### 4.3 交付验证闭环

```
                    ┌──────────────┐
                    │  Sub Cases   │
                    │  (Extended)  │
                    └──────┬───────┘
                           │
                    ┌──────▼───────┐
                    │ Build Image  │ ← Pipeline 1
                    │   Pipeline   │
                    └──────┬───────┘
                           │
                  ┌────────▼────────┐
                  │  Sanity Check   │ ← Pipeline 2
                  │    Pipeline     │
                  └────────┬────────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼─────┐ ┌───▼───┐ ┌─────▼─────┐
        │   PASS ✓  │ │ FAIL  │ │  ERROR    │
        │  有效交付  │ │ 修复后 │ │  排查后   │
        │           │ │ 重新提交│ │  重新提交  │
        └───────────┘ └───────┘ └───────────┘
```

**交付标准**：

- ✅ Build Image 成功构建且无 warning
- ✅ Sanity Check 全部通过
- ✅ 只有双重通过的 sub cases 才计入有效交付

## 5. Phase 4 — IT_Bench 研究与差异化分析

### 5.1 IT_Bench 概述

IT_Bench（IT Infrastructure Benchmark）是面向 **IT 基础设施运维场景**的 Agent 评测基准。与 Terminal-Bench2 侧重终端命令执行不同，IT_Bench 更关注复杂的 IT 运维工作流，包括故障诊断、服务部署、监控配置等。

### 5.2 TB2 vs IT_Bench 核心差异

| 对比维度       | Terminal-Bench2         | IT_Bench                                |
| -------------- | ----------------------- | --------------------------------------- |
| **评估焦点**   | 终端命令执行与系统操作  | IT 基础设施运维全流程                   |
| **任务粒度**   | 单一或短链命令任务      | 多步骤复合运维场景                      |
| **环境复杂度** | 单容器标准化环境        | 多组件、多服务交互环境                  |
| **构建方式**   | 标准 Docker 镜像        | 多容器编排 / VM 级别环境                |
| **验证逻辑**   | 输出匹配 + 状态检查     | 服务状态 + 日志分析 + 性能指标          |
| **脚本兼容性** | ❌ 不能与 IT_Bench 共用 | ❌ 需要独立的 Build & Sanity Check 脚本 |

### 5.3 关键发现

> **TB2 的 Build 和 Sanity Check 脚本不能直接复用于 IT_Bench。**

原因分析：

1. **环境构建差异**：IT_Bench 的 case 往往需要多个协作服务（如 DB + Web Server + Monitor），而 TB2 是单容器场景
2. **验证逻辑差异**：IT_Bench 的 pass/fail 判定涉及跨服务状态校验，远复杂于 TB2 的 stdout 比对
3. **生命周期差异**：IT_Bench 案例可能需要长时间运行来观测状态变化，TB2 案例通常为即时执行

## 6. 知识扩展：Terminal-Bench2 与 IT_Bench 技术纵深

### 6.1 AI Agent Benchmark 的行业背景

随着 LLM-powered Agent（如 GitHub Copilot、Cursor、Devin 等）的快速发展，行业急需标准化的评估基准来回答核心问题：

> **"这个 Agent 在真实场景中到底能做到什么程度？"**

主流 Benchmark 谱系：

| 基准                 | 侧重点               | 典型任务          |
| -------------------- | -------------------- | ----------------- |
| **SWE-bench**        | 软件工程（代码修复） | GitHub Issue 修复 |
| **HumanEval / MBPP** | 代码生成（函数级）   | 算法题实现        |
| **Terminal-Bench**   | 终端操作能力         | 系统命令执行      |
| **IT_Bench**         | IT 运维能力          | 基础设施故障排查  |
| **WebArena**         | Web 操作能力         | 浏览器交互任务    |
| **OSWorld**          | 操作系统级操作       | 跨应用桌面任务    |

### 6.2 Terminal-Bench2 技术深入

#### 6.2.1 设计哲学

TB2 的核心设计理念是**"终端即接口"** — 在 AI Agent 的实际应用中，终端是 Agent 与操作系统交互的主要通道。TB2 通过模拟真实终端场景，评估 Agent 的：

- **命令知识深度**：是否掌握命令的各种参数和组合方式
- **环境感知能力**：是否能正确理解当前系统状态
- **任务分解能力**：是否能将复杂目标拆解为可执行的命令序列
- **错误恢复能力**：遇到执行失败时是否能自主调整策略

#### 6.2.2 Seed Case 的分类体系

```
Terminal-Bench2 Seed Cases
├── System Administration
│   ├── 用户/组管理
│   ├── 权限控制
│   ├── 服务管理 (systemd / init)
│   └── 存储管理 (LVM / mount / fstab)
├── Networking
│   ├── 网络配置 (ip / nmcli)
│   ├── 防火墙规则 (iptables / firewalld)
│   ├── DNS 配置与排查
│   └── SSH 配置与隧道
├── Package & Environment
│   ├── 包管理 (apt / yum / dnf)
│   ├── 环境变量管理
│   ├── 编译工具链配置
│   └── 容器运行时 (Docker / Podman)
├── File & Text Processing
│   ├── 文件搜索 (find / locate)
│   ├── 文本处理 (sed / awk / grep)
│   ├── 归档与压缩
│   └── 文件系统操作
└── Scripting & Automation
    ├── Shell 脚本编写
    ├── Cron 定时任务
    ├── 进程管理 (ps / kill / nohup)
    └── 自动化工作流编排
```

#### 6.2.3 容器化评测环境

```dockerfile
# TB2 典型评测环境结构
FROM ubuntu:22.04
# 预装任务所需工具链
RUN apt-get update && apt-get install -y [required-packages]
# 设置初始状态（seed case 的前置条件）
COPY setup_scripts/ /opt/setup/
RUN /opt/setup/init.sh
# 注入评测 harness
COPY harness/ /opt/harness/
ENTRYPOINT ["/opt/harness/run.sh"]
```

### 6.3 IT_Bench 技术深入

#### 6.3.1 与 TB2 的本质差异

如果说 TB2 考的是"一个工程师在终端前能不能干好手头的活"，IT_Bench 考的是"一个 SRE 能不能在复杂生产环境中排查和解决问题"。

**关键区别**：

| 能力层级   | Terminal-Bench2  | IT_Bench              |
| ---------- | ---------------- | --------------------- |
| 命令执行   | ✅ 核心考察      | ✅ 基础能力           |
| 环境理解   | 单机环境         | 分布式多组件环境      |
| 因果推理   | 单步因果         | 多跳故障传播链        |
| 工具协同   | 单工具或简单管道 | 多工具 + 多服务联动   |
| 决策复杂度 | 路径较确定       | 开放式探索 + 假设验证 |

#### 6.3.2 IT_Bench 典型场景

```
场景示例: "Web 服务返回 502"

Agent 需要:
1. 检查 Nginx 状态和错误日志 → 发现 upstream timeout
2. 检查后端服务状态 → 发现服务运行但响应慢
3. 检查数据库连接 → 发现连接池耗尽
4. 检查数据库慢查询日志 → 定位到缺少索引的查询
5. 添加索引并验证服务恢复

涉及: Nginx + App Server + Database + 日志系统 的多组件协调
```

#### 6.3.3 对 Pipeline 的影响

```
IT_Bench Build Pipeline (与 TB2 不同):
├── 多容器编排 (docker-compose / K8s manifest)
├── 服务间依赖顺序管理
├── 故障注入脚本
├── 健康检查等待逻辑
└── 状态快照与回滚机制

IT_Bench Sanity Check Pipeline (与 TB2 不同):
├── 跨容器状态验证
├── 服务连通性检查
├── 故障注入验证（确认故障确实存在）
├── 修复后的多维度健康检查
└── 性能基线对比
```

## 8. 工作意义与价值反思

### 8.1 为什么这项工作重要

在 AI Agent 生态中，模型的能力往往被过度宣传或模糊定义。Core AI Training Team 的价值在于：

1. **建立客观标尺**：通过标准化评估，将模型能力从"感觉上不错"转化为"数据上第几名"
2. **扩大评测覆盖**：seed case → sub case 的扩展不是简单的数量堆叠，而是系统性地覆盖能力盲区
3. **保障交付质量**：Build + Sanity Check 双重门禁确保每一个 case 都是有效的、可靠的
4. **驱动模型进化**：评测发现的弱点直接反馈到模型训练管线，形成[评测→发现→优化→再评测]的正循环

### 8.2 个人成长与认知转变

> _"刚开始加入团队时，并没有过多意识，也不知道核心工作是什么。渐渐经历之后，发现这个团队就如其 title 一般——在模型 Agent 后方，对其做测试后期工作，查漏补缺，善于发现，总结细节，提炼优化方向。"_

这段认知转变本身就是从**执行者**到**理解者**的蜕变：

| 阶段             | 认知状态             | 工作模式                |
| ---------------- | -------------------- | ----------------------- |
| 初入团队         | 不清楚目标和工作流   | 被动接受任务            |
| Phase 1 完成后   | 理解评估工作的意义   | 能独立设计评估维度      |
| Phase 2-3 完成后 | 理解基准测试的工程化 | 能独立构建完整 pipeline |
| Phase 4 进行中   | 理解不同基准的差异性 | 能做技术选型和方案决策  |

### 8.3 方法论提炼

从这些工作中可以提炼出通用的方法论：

1. **先分析再扩展** — 不理解 seed case 的本质就做扩展，等于在沙上建楼
2. **Pipeline 即信任** — 自动化门禁是保障交付质量的唯一可靠方式
3. **差异即边界** — TB2 和 IT_Bench 的不兼容不是问题，而是清晰地划定了各自的适用边界
4. **从测试中学习** — 测试工作不只是"过不过"的判断，更是理解模型能力边界的最佳窗口
